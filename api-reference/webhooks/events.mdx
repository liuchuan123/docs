Every webhook PayMatrix delivers has a `type` field that tells you exactly what happened. Understanding each event type helps you build a robust integration that reacts correctly at each stage of the payment, refund, order, and settlement lifecycle. Below you'll find the complete catalog of events, a description of the standard payload structure every event shares, and guidance on verifying that incoming webhooks genuinely originated from PayMatrix.

## Event Types

| Event | Fires when |
|---|---|
| `payment.succeeded` | A payment is successfully captured |
| `payment.failed` | A payment attempt fails |
| `payment.cancelled` | A payment is cancelled |
| `refund.created` | A refund is initiated |
| `refund.succeeded` | A refund completes successfully |
| `refund.failed` | A refund fails |
| `order.expired` | An order expires without payment |
| `settlement.completed` | A daily settlement is processed |

## Webhook Payload Structure

Every event PayMatrix delivers shares the same outer envelope. The `data` object contains the resource that changed, and its fields vary by event type.

```json
{
  "id": "evt_EEEEEEEE",
  "type": "payment.succeeded",
  "created_at": "2024-01-15T10:30:05Z",
  "data": {
    "id": "pay_9Z8Y7X6W5V",
    "order_id": "ord_1A2B3C4D5E",
    "status": "succeeded",
    "amount": 5000,
    "currency": "USD"
  }
}
```

<Expandable title="Payload field descriptions">
  **`id`** — Unique identifier for this event, prefixed with `evt_`. Each event has a stable ID that you can use for idempotency checks if your endpoint receives the same event more than once.

  **`type`** — The event type string (e.g., `payment.succeeded`). Always check this field before processing to ensure your handler is responding to the right event.

  **`created_at`** — ISO 8601 timestamp of when the event was generated by PayMatrix.

  **`data`** — The full resource object associated with the event. For payment events this is the Payment object; for refund events it's the Refund object, and so on.
</Expandable>

## Signature Verification

PayMatrix signs every webhook request with an HMAC-SHA256 signature computed from the raw request body using your webhook secret. The signature is sent in the `X-PayMatrix-Signature` header as a hex-encoded string. You should **always** verify this signature before processing any event payload to ensure the request genuinely originated from PayMatrix and has not been tampered with.

<CodeGroup>
```javascript Node.js
const crypto = require('crypto');

app.post('/webhooks/paymatrix', express.raw({ type: 'application/json' }), (req, res) => {
  const signature = req.headers['x-paymatrix-signature'];
  const secret = process.env.PAYMATRIX_WEBHOOK_SECRET;

  const expected = crypto
    .createHmac('sha256', secret)
    .update(req.body)
    .digest('hex');

  if (!crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expected))) {
    return res.status(400).send('Invalid signature');
  }

  const event = JSON.parse(req.body);
  // Handle event...
  res.sendStatus(200);
});
```

```python Python
import hmac
import hashlib
import os
from flask import Flask, request, abort

app = Flask(__name__)

@app.route('/webhooks/paymatrix', methods=['POST'])
def handle_webhook():
    signature = request.headers.get('X-PayMatrix-Signature')
    secret = os.environ['PAYMATRIX_WEBHOOK_SECRET'].encode()
    raw_body = request.get_data()

    expected = hmac.new(secret, raw_body, hashlib.sha256).hexdigest()

    if not hmac.compare_digest(signature, expected):
        abort(400, 'Invalid signature')

    event = request.get_json(force=True)
    # Handle event...
    return '', 200
```
</CodeGroup>

<Note>
  Always use a **timing-safe comparison** (e.g., `crypto.timingSafeEqual` in Node.js or `hmac.compare_digest` in Python) when validating the signature. Regular string equality checks are vulnerable to timing attacks.
</Note>

## Retry Policy

If your endpoint returns a non-`2xx` HTTP status code or does not respond within 10 seconds, PayMatrix will automatically retry delivery up to **3 times** using exponential backoff:

| Attempt | Delay after previous failure |
|---|---|
| 1st retry | 1 minute |
| 2nd retry | 5 minutes |
| 3rd retry | 30 minutes |

After three failed attempts, the event is marked as undelivered and no further retries occur. You can view undelivered events in your PayMatrix dashboard and replay them manually if needed.

<Tip>
  Respond with a `200` status immediately upon receiving a webhook and then process the event asynchronously (e.g., via a job queue). This prevents slow internal processing from causing delivery timeouts and unnecessary retries.
</Tip>
