Every request you make to the PayMatrix API must be authenticated using an API key. PayMatrix uses the standard Bearer token scheme — you include your key in the `Authorization` header of each HTTP request, and PayMatrix validates it before processing anything. There are no sessions, cookies, or OAuth flows required for server-to-server API calls.

## API key types

PayMatrix issues two distinct key types, each targeting a separate environment. You can find both in the dashboard under **Settings → API Keys**.

<CardGroup cols={2}>
  <Card title="Test keys" icon="flask">
    Prefixed with `pm_test_`. All requests made with a test key are routed to the PayMatrix sandbox — a fully functional environment that mirrors production but never moves real money.
  </Card>
  <Card title="Live keys" icon="bolt">
    Prefixed with `pm_live_`. Requests made with a live key process real payments and trigger real fund movements. Use these only in your production environment.
  </Card>
</CardGroup>

<Note>
  Your test and live environments are completely isolated. Orders, payments, customers, and webhook configurations created in the sandbox are never visible in production, and vice versa. This means you can build and test freely without any risk of polluting your live data.
</Note>

## Making authenticated requests

Pass your API key as a Bearer token in the `Authorization` header on every request. There is no additional token exchange or session initialization step required.

```bash
curl https://api.paymatrix.com/v1/payments \
  -H "Authorization: Bearer pm_test_xxxxxxxxxxxx"
```

Replace `pm_test_xxxxxxxxxxxx` with your actual key copied from the dashboard. The same pattern applies to every endpoint in the PayMatrix API, regardless of the HTTP method.

<Tabs>
  <Tab title="cURL">
    ```bash
    curl -X POST https://api.paymatrix.com/v1/orders \
      -H "Authorization: Bearer pm_test_xxxxxxxxxxxx" \
      -H "Content-Type: application/json" \
      -d '{ "amount": 5000, "currency": "USD", "description": "Order #1001" }'
    ```
  </Tab>
  <Tab title="Node.js">
    ```javascript
    const response = await fetch("https://api.paymatrix.com/v1/orders", {
      method: "POST",
      headers: {
        "Authorization": "Bearer pm_test_xxxxxxxxxxxx",
        "Content-Type": "application/json",
      },
      body: JSON.stringify({
        amount: 5000,
        currency: "USD",
        description: "Order #1001",
      }),
    });
    ```
  </Tab>
  <Tab title="Python">
    ```python
    import requests

    response = requests.post(
        "https://api.paymatrix.com/v1/orders",
        headers={
            "Authorization": "Bearer pm_test_xxxxxxxxxxxx",
            "Content-Type": "application/json",
        },
        json={
            "amount": 5000,
            "currency": "USD",
            "description": "Order #1001",
        },
    )
    ```
  </Tab>
</Tabs>

## Security best practices

Keeping your API keys secure is critical. A leaked live key allows anyone to create payments, issue refunds, and access your transaction data.

<Warning>
  Never expose your API keys in client-side code (JavaScript bundles, mobile apps, or browser requests), public repositories, or logs. If a key is compromised, revoke it immediately from **Settings → API Keys** and replace it with a newly generated key.
</Warning>

Follow these practices to keep your keys safe:

- **Store keys in environment variables.** Never hard-code a key directly in your source files. Use environment variables or a secrets manager like AWS Secrets Manager, HashiCorp Vault, or Doppler.
- **Keep test and live keys separate.** Use your test key in all development and CI environments. Gate access to the live key strictly to your production deployment.
- **Restrict access by team member.** Only share live keys with engineers and systems that genuinely need them. Use role-based access in the dashboard to limit who can view or generate keys.
- **Audit regularly.** Periodically review the keys listed in your dashboard and revoke any that are no longer in use.

## Authentication failures

If your request includes a missing, malformed, or revoked API key, PayMatrix returns an HTTP `401 Unauthorized` response with a JSON error body.

```json
{
  "error": {
    "code": "authentication_required",
    "message": "No valid API key provided."
  }
}
```

Common causes of `401` errors include:

- Omitting the `Authorization` header entirely
- Using a key that has been revoked from the dashboard
- Passing a test key to a live endpoint, or vice versa (the environments are isolated)
- Typographical errors in the key value (ensure there are no leading or trailing spaces)

## Rotating your API keys

You should rotate your API keys periodically and immediately any time you suspect a key has been exposed. To rotate a key without downtime:

<Steps>
  <Step title="Generate a new key">
    In the dashboard, go to **Settings → API Keys** and click **Generate New Key**. A new key is created alongside your existing one — both are active at this point.
  </Step>
  <Step title="Update your integration">
    Deploy the new key to all environments and services that use the old key. Verify that requests are succeeding with the new key before proceeding.
  </Step>
  <Step title="Revoke the old key">
    Once you've confirmed everything is working with the new key, return to **Settings → API Keys** and revoke the old key. Revocation is immediate and permanent.
  </Step>
</Steps>

<Tip>
  When rotating keys as part of a security incident response, generate the new key and revoke the old key in rapid succession to minimize the window of exposure. Deploy the new key to production before revoking the old one to avoid downtime.
</Tip>
