PMP Payment 提供两种收银台集成方案，商户可根据需求选择。

## 方案对比

| 特性               | 托管收银台                         | 自建收银台                         |
| ------------------ | ---------------------------------- | ---------------------------------- |
| 开发工作量         | 极少，一行跳转即可                 | 需要前端开发                       |
| UI 定制化          | 平台统一风格，商户可定制 Logo      | 完全自主控制 UI 和交互             |
| PCI 合规           | 平台负责，商户无需关心             | 需自行确保 SAQ A-EP 合规           |
| 支付方式支持       | 全部支持                           | 根据接入的 SDK 决定                |
| 3DS 认证           | 自动处理                           | 需自行调用 Stripe.js 等 SDK 处理   |
| 适用场景           | 大部分商户                         | 有深度定制需求的商户               |

<Tip>
  推荐大多数商户使用**托管收银台**方案，开发成本最低，且无需担心 PCI 合规问题。
</Tip>

## 托管收银台

商户创建支付订单后，将用户重定向到平台提供的收银台地址。

```
商户网站 → 创建支付订单 → 跳转 pay_url → 托管收银台 → 支付完成 → 跳回商户网站
```

### 集成步骤

1. 商户服务端调用 `POST /merchant/payment/create` 获取 `pay_url`
2. 将用户浏览器重定向到 `pay_url`
3. 用户在托管收银台选择支付方式并完成支付
4. 支付完成后自动跳转回 `redirect_url`
5. 同时，平台推送 Webhook 确认支付结果

详见 [托管收银台集成](/checkout/hosted)。

## 自建收银台

商户在自己的网站/应用中构建支付表单，直接调用平台 API 完成支付。

```
商户前端 → 获取支付方式 → 初始化支付 → 渲染 SDK 元素 → 提交支付 → 跳回商户网站
```

### 集成步骤

1. 调用 `GET /api/web/checkout/payment-methods?orderId=xxx` 获取可用的支付方式
2. 调用 `POST /api/web/checkout/init` 初始化支付
3. 根据返回的 `paymentRenderType` 渲染对应的支付组件
4. 用户填写支付信息后，调用 `POST /api/web/payment/submit` 提交支付
5. 根据返回结果处理 3DS 重定向或直接跳转

### 支持的支付渲染方式

| 渲染类型          | 说明                                           | 适用渠道     |
| ----------------- | ---------------------------------------------- | ------------ |
| `STRIPE_ELEMENT`  | 使用 Stripe.js Elements SDK 渲染卡号输入表单   | Stripe       |
| `COMPONENT_SDK`   | 使用渠道自有 SDK（如 Antom SDK）渲染支付组件   | Antom/Alipay |
| `OWN_FORM`        | 商户完全自建卡号表单，直接传给平台             | LianLian 等  |
| `REDIRECT`        | 跳转到第三方支付页面（如网银页面）             | 银行直连等   |

详见 [自建收银台集成](/checkout/custom)。

## 安全要点

- 无论哪种方案，支付成功均应以 **Webhook 通知**为准，不可信任前端回调
- 自建收银台方式下，涉及卡号的表单需确保 PCI 合规
- 始终使用 HTTPS
