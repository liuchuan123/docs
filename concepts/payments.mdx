A **Payment** is the core transaction object in PayMatrix. When your customer submits their payment details, PayMatrix creates a Payment record that tracks every stage of the transaction — from the initial authorization request through to final settlement. Each Payment is linked to an [Order](/concepts/orders) and carries a full audit trail, including the payment method used, the amount charged, and timestamps for every status transition.

## Payment Lifecycle

Payments move through a defined set of states from the moment they are initiated. Understanding these states helps you build reliable payment flows and handle edge cases gracefully.

| State | Description |
|---|---|
| `pending` | The payment has been created but authorization has not yet been attempted. |
| `processing` | PayMatrix has submitted the authorization request to the payment network and is awaiting a response. |
| `succeeded` | The payment was authorized and funds have been captured successfully. |
| `failed` | Authorization or capture was declined. No funds were collected from your customer. |
| `cancelled` | The payment was cancelled before capture, either by your application or automatically when the parent order expired. |

<Note>
  A payment in `failed` state does not automatically retry. You should create a new payment on the same order — or prompt your customer to try a different payment method — to recover from a failure.
</Note>

## Supported Payment Methods

PayMatrix supports three categories of payment method. You specify the `method` field when creating a payment.

<CardGroup cols={3}>
  <Card title="Card" icon="credit-card">
    Visa, Mastercard, American Express, and Discover. Both credit and debit cards are supported. Card details are tokenized and never stored on your servers.
  </Card>
  <Card title="Bank Transfer" icon="building-columns">
    Direct account-to-account transfers via ACH (US) and SEPA (EU). Ideal for large transactions and recurring billing.
  </Card>
  <Card title="Digital Wallet" icon="wallet">
    Apple Pay, Google Pay, and PayPal. Customers authenticate using their device or wallet credentials for a faster checkout experience.
  </Card>
</CardGroup>

## The Payment Object

Below is a representative Payment object returned by the PayMatrix API. All monetary values are expressed as integers in the **smallest currency unit** — for USD, that means cents. An `amount` of `5000` equals $50.00.

```json
{
  "id": "pay_9Z8Y7X6W5V",
  "order_id": "ord_1A2B3C4D5E",
  "status": "succeeded",
  "amount": 5000,
  "currency": "USD",
  "method": "card",
  "card": {
    "brand": "visa",
    "last4": "4242",
    "exp_month": 12,
    "exp_year": 2027
  },
  "created_at": "2024-01-15T10:30:05Z",
  "settled_at": null
}
```

<Accordion title="Payment object field reference">

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique identifier for the payment, prefixed with `pay_`. |
| `order_id` | string | The ID of the Order this payment belongs to. |
| `status` | string | Current lifecycle state. See [Payment Lifecycle](#payment-lifecycle). |
| `amount` | integer | Charge amount in the smallest currency unit (e.g. cents for USD). |
| `currency` | string | Three-letter ISO 4217 currency code. |
| `method` | string | Payment method type: `card`, `bank_transfer`, or `digital_wallet`. |
| `card` | object | Present when `method` is `card`. Contains `brand`, `last4`, `exp_month`, and `exp_year`. |
| `created_at` | string | ISO 8601 timestamp of when the payment was created. |
| `settled_at` | string \| null | ISO 8601 timestamp of when the payment was included in a settlement. `null` until settled. |

</Accordion>

## Multi-Currency Payments

PayMatrix processes payments in any of the supported currencies your merchant account is configured for. Regardless of currency, **amounts are always expressed as integers in the smallest unit** of that currency.

<Tabs>
  <Tab title="Zero-decimal currencies">
    Some currencies — such as JPY and KRW — have no minor unit. For these, the `amount` value represents the full currency amount directly. A charge of ¥500 is submitted as `"amount": 500`.
  </Tab>
  <Tab title="Two-decimal currencies">
    Currencies like USD, EUR, and GBP use cents (or equivalent) as their smallest unit. A charge of $50.00 is submitted as `"amount": 5000`.
  </Tab>
  <Tab title="Three-decimal currencies">
    Currencies such as KWD and BHD use fils as their smallest unit (1/1000). A charge of 5.000 KWD is submitted as `"amount": 5000`.
  </Tab>
</Tabs>

<Tip>
  Always store and transmit amounts as integers. Avoid floating-point arithmetic when calculating totals — use integer math throughout your payment flow to prevent rounding errors.
</Tip>
