When something goes wrong with an API request, PayMatrix returns a structured error response that tells you exactly what failed and why. Understanding how to read these responses — and knowing what each error code means — will help you build a resilient integration that handles failures gracefully and surfaces clear messages to your users.

## Error Response Format

Every error response from the PayMatrix API uses a consistent JSON envelope. The top-level `error` object contains four fields that together give you everything you need to diagnose and resolve the issue.

```json
{
  "error": {
    "code": "card_declined",
    "message": "The card was declined by the issuing bank.",
    "param": "card.number",
    "request_id": "req_XXXXXXXX"
  }
}
```

| Field | Description |
|---|---|
| `code` | A machine-readable string identifying the error type. Use this in your application logic to determine how to respond. |
| `message` | A human-readable description of the error. This is suitable for logging but should be translated into user-friendly copy before being shown to customers. |
| `param` | The specific request parameter that caused the error, if applicable. Not present on all errors. |
| `request_id` | A unique identifier for this API request. Always include this when contacting PayMatrix support. |

## HTTP Status Codes

PayMatrix uses standard HTTP status codes to indicate the general category of a response. Check the status code first, then inspect the `error.code` field for granular detail.

| Status | Meaning |
|---|---|
| `200` | **Success** — the request was processed successfully. |
| `201` | **Resource created** — a new resource (e.g., payment or order) was created successfully. |
| `400` | **Bad request** — the request body is malformed or missing required fields. Check your request structure. |
| `401` | **Unauthorized** — no valid API key was provided or the key has been revoked. |
| `403` | **Forbidden** — the API key is valid but does not have permission to perform this action. |
| `404` | **Resource not found** — the requested resource ID does not exist in your account. |
| `409` | **Conflict** — the request conflicts with the current state, such as a duplicate submission. |
| `422` | **Unprocessable entity** — the request is well-formed but fails validation (e.g., an invalid amount). |
| `429` | **Rate limit exceeded** — you have sent too many requests in a short period. Slow down and retry. |
| `500` | **Internal server error** — an unexpected error occurred on PayMatrix's side. Contact support. |

## Common Error Codes

The table below lists the most frequently encountered `error.code` values, what they mean, and the recommended action for each.

| Code | Description | How to Fix |
|---|---|---|
| `authentication_required` | No valid API key was provided with the request. | Include an `Authorization: Bearer <your_api_key>` header on every request. |
| `invalid_api_key` | The API key is malformed, expired, or has been revoked. | Verify the key in **Settings → API Keys** on your dashboard and generate a new one if needed. |
| `card_declined` | The card was rejected by the issuing bank. | Ask the customer to use a different card or contact their bank for details. |
| `insufficient_funds` | The card does not have enough balance to cover the payment. | Ask the customer to use a different card or payment method. |
| `expired_card` | The card's expiry date has passed. | Ask the customer to update their card details with a valid card. |
| `invalid_cvc` | The CVC code provided does not match the card. | Ask the customer to re-enter their CVC carefully. |
| `order_not_found` | The specified order ID does not exist under your account. | Verify the order was created successfully before attempting to pay it. |
| `order_already_paid` | The order already has a successfully completed payment. | Check the order status before creating a new payment to avoid double-charging. |
| `amount_too_small` | The payment amount is below the minimum threshold. | Ensure the amount is at least `50` (in the smallest currency unit, e.g., 50 cents). |
| `amount_too_large` | The payment amount exceeds the maximum limit for your account. | Contact PayMatrix support to discuss higher transaction limits. |
| `rate_limit_exceeded` | Your integration has sent too many requests in a short window. | Implement exponential backoff and honour the `Retry-After` response header. |
| `idempotency_conflict` | The same `Idempotency-Key` was reused with different request parameters. | Use a unique idempotency key for each distinct request. |

## Rate Limits

PayMatrix enforces rate limits to ensure fair usage and platform stability across all merchants.

<Note>
  The default rate limit is **100 requests per minute** per API key. Sandbox and production keys have separate rate limit buckets.
</Note>

When you exceed the limit, the API returns an HTTP `429` response. The response includes a `Retry-After` header indicating the number of seconds you must wait before making another request:

```
HTTP/1.1 429 Too Many Requests
Retry-After: 12
```

To handle rate limits gracefully in your integration, implement **exponential backoff with jitter**: wait for the `Retry-After` duration, then retry the request. Avoid immediately retrying in a tight loop, as this will continue to return `429` responses and delay your recovery.

```javascript
async function requestWithBackoff(fn, maxRetries = 5) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const response = await fn();
    if (response.status !== 429) return response;

    const retryAfter = parseInt(response.headers.get("Retry-After") || "1", 10);
    const jitter = Math.random() * 1000; // up to 1 second of jitter
    await new Promise((resolve) => setTimeout(resolve, retryAfter * 1000 + jitter));
  }
  throw new Error("Max retries exceeded");
}
```

If you consistently approach the rate limit, contact [support@paymatrix.io](mailto:support@paymatrix.io) to discuss a higher limit for your account.

## Idempotency Keys

Network failures and timeouts can leave you uncertain about whether a request succeeded. To safely retry a `POST` request without risking a duplicate payment, use an **idempotency key**.

Include the `Idempotency-Key` header with a unique string (such as a UUID) on every `POST` request:

```bash
curl https://api.paymatrix.io/v1/payments \
  -H "Authorization: Bearer sk_live_••••" \
  -H "Idempotency-Key: 4b3f9a12-7c2e-4d8a-b1f3-9e0d6c5a2b87" \
  -H "Content-Type: application/json" \
  -d '{
    "order_id": "ord_abc123",
    "amount": 2500,
    "currency": "usd"
  }'
```

<Warning>
  If you reuse an `Idempotency-Key` with **different parameters**, the API will return an `idempotency_conflict` error. Always use a fresh key for each distinct logical operation, and reuse the same key only when retrying an identical request.
</Warning>

PayMatrix stores idempotency keys for **24 hours**. Within that window, any repeat request with the same key and matching parameters will return the cached response from the first successful attempt — no duplicate payment will be created.

## Contacting Support

If you encounter an error you cannot resolve — particularly any `500` status code or unexpected `403` response — our support team is here to help.

<Tip>
  Always include the `request_id` from the error response when contacting PayMatrix support. This value (e.g., `req_XXXXXXXX`) lets our engineers locate the exact request in our logs within seconds, dramatically speeding up diagnosis and resolution.
</Tip>

Reach the support team at [support@paymatrix.io](mailto:support@paymatrix.io) or via the live chat widget in the dashboard. For security incidents such as a suspected compromised API key, email [security@paymatrix.io](mailto:security@paymatrix.io) directly.
