PayMatrix makes it straightforward to accept payments across cards, bank transfers, and digital wallets through a unified REST API. The flow follows two core steps — creating an order to represent what your customer is buying, then submitting a payment against that order. This guide walks you through the complete integration from your first API call to handling the final payment response.

<Steps>

  <Step title="Create an Order">
    Every payment in PayMatrix begins with an order. Send a `POST` request to `/v1/orders` with the amount (in the smallest currency unit, e.g. cents for USD), currency, and an optional description. The returned `order_id` ties together the checkout session and the eventual payment record.

    <CodeGroup>
      ```bash Request
      curl -X POST https://api.paymatrix.com/v1/orders \
        -H "Authorization: Bearer pm_test_xxxxxxxxxxxx" \
        -H "Content-Type: application/json" \
        -d '{"amount": 9900, "currency": "USD", "description": "Pro subscription"}'
      ```

      ```json Response
      {
        "id": "ord_1A2B3C4D5E",
        "status": "pending",
        "amount": 9900,
        "currency": "USD"
      }
      ```
    </CodeGroup>

    <Tip>
      Store the `id` from the response — you'll pass it as `order_id` when you submit the payment in a later step.
    </Tip>
  </Step>

  <Step title="Display Checkout to Your Customer">
    Before collecting any payment details, you need to present a checkout experience to your customer. PayMatrix's hosted checkout fields handle sensitive data capture in a PCI-compliant iframe, so card numbers never touch your servers.

    Refer to the [Hosted Checkout integration guide](/integration/checkout) for full setup instructions, including how to initialize the checkout widget with your `order_id` and render the appropriate payment method fields.
  </Step>

  <Step title="Submit a Payment">
    Once your customer has entered their payment details, submit a `POST` request to `/v1/payments`. Pass the `order_id` from Step 1 along with the payment `method` and the corresponding method-specific payload.

    <Warning>
      Never pass raw card numbers directly from your own servers in production. Doing so brings your infrastructure into PCI DSS scope and creates significant security risk. Use [PayMatrix Hosted Fields](/integration/checkout) to tokenize card data before it reaches your backend.
    </Warning>

    The example below shows a card payment. Use the tabs to compare the request body for bank transfers and digital wallets.

    <CodeGroup>
      ```bash Card
      curl -X POST https://api.paymatrix.com/v1/payments \
        -H "Authorization: Bearer pm_test_xxxxxxxxxxxx" \
        -H "Content-Type: application/json" \
        -d '{
          "order_id": "ord_1A2B3C4D5E",
          "method": "card",
          "card": {
            "number": "4242424242424242",
            "exp_month": 12,
            "exp_year": 2027,
            "cvc": "123"
          }
        }'
      ```

      ```bash Bank Transfer
      curl -X POST https://api.paymatrix.com/v1/payments \
        -H "Authorization: Bearer pm_test_xxxxxxxxxxxx" \
        -H "Content-Type: application/json" \
        -d '{
          "order_id": "ord_1A2B3C4D5E",
          "method": "bank_transfer",
          "bank_transfer": {
            "account_number": "000123456789",
            "routing_number": "110000000",
            "account_type": "checking"
          }
        }'
      ```

      ```bash Digital Wallet
      curl -X POST https://api.paymatrix.com/v1/payments \
        -H "Authorization: Bearer pm_test_xxxxxxxxxxxx" \
        -H "Content-Type: application/json" \
        -d '{
          "order_id": "ord_1A2B3C4D5E",
          "method": "wallet",
          "wallet": {
            "type": "paymatrix_wallet",
            "token": "wlt_tok_xxxxxxxxxxxx"
          }
        }'
      ```
    </CodeGroup>
  </Step>

  <Step title="Handle the Payment Response">
    A successful request to `/v1/payments` returns a payment object. Always inspect the `status` field — do not assume that a `200 OK` HTTP response means the payment was captured.

    ```json
    {
      "id": "pay_9Z8Y7X6W5V",
      "status": "succeeded",
      "amount": 9900,
      "currency": "USD"
    }
    ```

    <CardGroup cols={2}>
      <Card title="succeeded" icon="circle-check">
        The payment was captured successfully. You can safely fulfil the order and show a confirmation to your customer.
      </Card>
      <Card title="failed" icon="circle-xmark">
        The payment was declined or encountered an error. Check the `failure_code` and `failure_message` fields for details, then prompt your customer to retry with a different method.
      </Card>
    </CardGroup>

    Other statuses you may encounter include `processing` (common for bank transfers, which settle asynchronously) and `requires_action` (when 3D Secure authentication is needed). Handle these states by polling the payment resource or listening for webhook events.
  </Step>

  <Step title="Listen for Webhook Events">
    Payment outcomes for asynchronous methods — such as bank transfers — are delivered via webhooks rather than synchronously in the API response. Even for card payments, relying on webhooks for fulfilment is more resilient than trusting only the initial response, since network interruptions can prevent your client from receiving it.

    Set up your webhook endpoint to listen for `payment.succeeded` and `payment.failed` events. See the [Webhooks guide](/guides/webhooks) for full instructions on registering your endpoint and verifying event signatures.
  </Step>

</Steps>
