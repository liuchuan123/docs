Refunds in PayMatrix let you return funds to a customer for any captured payment — whether you need to reverse the full amount or issue a partial credit. Refunds are initiated against an existing payment ID, flow through a short lifecycle of states, and typically appear on your customer's statement within 5–10 business days depending on their card network or bank.

## How Refunds Work

To issue a refund, send a `POST` request to `/v1/payments/{id}/refund`, where `{id}` is the `id` of the original payment you want to reverse. PayMatrix will create a new refund object linked to that payment and begin the return process with the relevant payment network.

<Note>
  Refunds are deducted from your next scheduled settlement payout rather than charged as a separate transaction. Keep this in mind when reconciling your account balance.
</Note>

## Full vs. Partial Refunds

By default, omitting the `amount` field issues a full refund for the original payment amount. To issue a partial refund, include the `amount` field set to the value you want to return (in the smallest currency unit).

<CodeGroup>
  ```bash Full Refund
  curl -X POST https://api.paymatrix.com/v1/payments/pay_9Z8Y7X6W5V/refund \
    -H "Authorization: Bearer pm_test_xxxxxxxxxxxx" \
    -H "Content-Type: application/json" \
    -d '{}'
  ```

  ```bash Partial Refund
  curl -X POST https://api.paymatrix.com/v1/payments/pay_9Z8Y7X6W5V/refund \
    -H "Authorization: Bearer pm_test_xxxxxxxxxxxx" \
    -H "Content-Type: application/json" \
    -d '{
      "amount": 2500,
      "reason": "customer_request"
    }'
  ```
</CodeGroup>

A successful request returns a refund object:

```json
{
  "id": "ref_RRRRRRR",
  "payment_id": "pay_9Z8Y7X6W5V",
  "status": "pending",
  "amount": 2500,
  "currency": "USD",
  "reason": "customer_request",
  "created_at": "2024-01-15T11:00:00Z"
}
```

<Warning>
  The total refunded amount across all refunds for a payment cannot exceed the original captured amount. Attempts to refund beyond this limit will return a `422 Unprocessable Entity` error.
</Warning>

## Refund Reason Codes

Including a `reason` helps with reporting and dispute management. PayMatrix accepts the following values for the `reason` field:

| Code | When to Use |
|---|---|
| `customer_request` | The customer asked for a refund (e.g. changed mind, unused service) |
| `duplicate` | The payment was processed more than once in error |
| `fraudulent` | The payment is suspected or confirmed to be fraudulent |

The `reason` field is optional but recommended — it is surfaced in your dashboard reports and settlement exports.

## Refund Lifecycle

Refunds move through the following states after creation. You can track state changes by polling `GET /v1/payments/{id}/refund/{refund_id}` or by listening for `refund.*` webhook events.

```
pending → processing → succeeded
                    ↘ failed
```

<CardGroup cols={3}>
  <Card title="pending" icon="clock">
    The refund has been created and queued. No funds have moved yet.
  </Card>
  <Card title="processing" icon="arrows-rotate">
    The refund has been submitted to the payment network and is in transit.
  </Card>
  <Card title="succeeded" icon="circle-check">
    The funds have been returned. The customer should see the credit within 5–10 business days.
  </Card>
</CardGroup>

<Tip>
  If a refund reaches `failed` status, retrieve the refund object and inspect the `failure_reason` field. Common causes include the original payment method being closed or the issuing bank rejecting the return. In these cases, you may need to issue the refund via an alternative method such as a bank transfer or store credit.
</Tip>
