Webhooks allow your application to react to payment events the moment they happen, without polling the PayMatrix API. When an event occurs — such as a payment succeeding or a refund being created — PayMatrix sends an HTTP `POST` request to a URL you specify, carrying a signed JSON payload that describes the event. This guide walks you through registering an endpoint, handling incoming payloads, and verifying that each request genuinely came from PayMatrix.

<Steps>

  <Step title="Register Your Endpoint">
    You can register a webhook endpoint through the PayMatrix Dashboard or directly via the API.

    **Via the Dashboard:** Navigate to **Settings → Webhooks → Add Endpoint**, enter your publicly accessible HTTPS URL, and select the event types you want to subscribe to.

    **Via the API:** Send a `POST` request to `/v1/webhooks` with your endpoint URL and the list of event types to subscribe to.

    ```bash
    curl -X POST https://api.paymatrix.com/v1/webhooks \
      -H "Authorization: Bearer pm_test_xxxxxxxxxxxx" \
      -H "Content-Type: application/json" \
      -d '{
        "url": "https://yourapp.com/webhooks/paymatrix",
        "events": [
          "payment.succeeded",
          "payment.failed",
          "refund.created"
        ]
      }'
    ```

    <Note>
      Your webhook endpoint must be reachable over HTTPS. PayMatrix will not deliver events to plain HTTP URLs in production mode.
    </Note>
  </Step>

  <Step title="Handle the Event Payload">
    When a subscribed event fires, PayMatrix sends a `POST` request to your endpoint with a JSON body structured as follows. The top-level `type` field tells you what kind of event occurred, and the `data` object contains the full resource that changed.

    ```json
    {
      "id": "evt_EEEEEEEE",
      "type": "payment.succeeded",
      "created_at": "2024-01-15T10:30:05Z",
      "data": {
        "id": "pay_9Z8Y7X6W5V",
        "order_id": "ord_1A2B3C4D5E",
        "status": "succeeded",
        "amount": 5000,
        "currency": "USD"
      }
    }
    ```

    In your handler, switch on the `type` field to route each event to the appropriate business logic. Treat the `data` object as the source of truth for the resource's current state rather than making a follow-up API call unless you need fields not present in the payload.

    <Tip>
      Make your event handlers **idempotent** — PayMatrix may deliver the same event more than once under certain retry conditions. Use the event `id` to deduplicate processing on your side.
    </Tip>
  </Step>

  <Step title="Verify the Signature">
    Every webhook request includes an `X-PayMatrix-Signature` header containing an HMAC-SHA256 digest of the raw request body, signed with your endpoint's secret key. Always verify this signature before processing the event — it guarantees the payload came from PayMatrix and has not been tampered with.

    Your webhook secret is shown once when you create the endpoint in the Dashboard. Store it securely as an environment variable.

    <Warning>
      Verify the signature against the **raw** request body bytes before any JSON parsing. Parsing and re-serialising the body may alter whitespace or key ordering, causing legitimate signatures to fail verification.
    </Warning>

    ```javascript
    const crypto = require('crypto');

    function verifySignature(payload, signature, secret) {
      const expected = crypto
        .createHmac('sha256', secret)
        .update(payload)
        .digest('hex');
      return crypto.timingSafeEqual(
        Buffer.from(signature),
        Buffer.from(expected)
      );
    }
    ```

    Call this function in your request handler, passing the raw body buffer, the value of the `X-PayMatrix-Signature` header, and your stored webhook secret. If the function returns `false`, respond with `400 Bad Request` and do not process the event.
  </Step>

  <Step title="Respond with HTTP 200">
    After validating the signature and queuing the event for processing, your endpoint must return a `2xx` status code — ideally `200 OK` — within **5 seconds**. PayMatrix treats any non-`2xx` response, or a timeout, as a delivery failure and will retry the event.

    Retry schedule:
    - **Attempt 2:** 30 seconds after the first failure
    - **Attempt 3:** 5 minutes after the second failure
    - **Attempt 4:** 30 minutes after the third failure

    After four failed attempts, the event is marked as undelivered and you can manually replay it from the Dashboard under **Settings → Webhooks → Event Log**.

    <Tip>
      To keep your response time well under the 5-second limit, acknowledge the webhook immediately by returning `200` and then process the event asynchronously using a background job queue.
    </Tip>
  </Step>

</Steps>

## Event Types Reference

PayMatrix emits the following event types. Subscribe only to the events relevant to your integration to reduce noise in your handler.

| Event | Description |
|---|---|
| `payment.succeeded` | A payment was captured successfully |
| `payment.failed` | A payment attempt was declined or encountered an error |
| `payment.cancelled` | A pending payment was cancelled before capture |
| `refund.created` | A refund was initiated against a payment |
| `refund.succeeded` | A refund completed and funds were returned to the customer |
| `refund.failed` | A refund could not be completed |
| `settlement.completed` | A settlement payout has been sent to your bank account |
| `order.expired` | An order reached its expiry time without a completed payment |
