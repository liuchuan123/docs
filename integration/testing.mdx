The PayMatrix sandbox is a fully isolated environment that mirrors production in every way — same API endpoints, same request/response shapes, same webhook events — but processes no real money and makes no connections to card networks. You can simulate successful payments, declines, 3D Secure challenges, refunds, and more, giving you full confidence in your integration before going live.

## Switching to the Sandbox

To use the sandbox, replace your live API keys with your **test keys**. Test keys always begin with `pm_test_` and are found in your dashboard under **Settings → API Keys → Test**.

```bash
# Sandbox request — note the pm_test_ key
curl -X POST https://api.paymatrix.com/v1/orders \
  -H "Authorization: Bearer pm_test_xxxxxxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "amount": 4999,
    "currency": "usd",
    "description": "Test Order #1234"
  }'
```

<Warning>
  Never use live (`pm_live_`) keys in a development or staging environment. Doing so creates real charges and real payouts against your merchant account.
</Warning>

## Test Card Numbers

Use the card numbers below to simulate specific payment outcomes in the sandbox. For all test cards, you can use any future expiry date, any three-digit CVC, and any billing ZIP code.

| Card Number | Behavior |
|---|---|
| 4242 4242 4242 4242 | Payment succeeds |
| 4000 0000 0000 0002 | Card declined |
| 4000 0000 0000 9995 | Insufficient funds |
| 4000 0000 0000 3220 | 3D Secure required |
| 4000 0000 0000 0069 | Expired card |

<Note>
  These card numbers are only valid in the sandbox. Submitting them with a live API key will result in a declined or invalid card error.
</Note>

## What You Can Simulate

The sandbox supports the full range of PayMatrix features, so you can exercise every part of your integration without risking real funds.

<CardGroup cols={3}>
  <Card title="Refunds" icon="arrow-uturn-left">
    Issue full or partial refunds via `POST /v1/refunds` exactly as you would in production. Refunded amounts are reflected immediately in sandbox transaction history.
  </Card>
  <Card title="Webhooks" icon="bolt">
    All webhook events (`payment.succeeded`, `payment.failed`, `refund.created`, etc.) fire in the sandbox. Use [webhook.site](https://webhook.site) or a local tunnel like ngrok to inspect payloads during development.
  </Card>
  <Card title="Settlement" icon="building-columns">
    Sandbox settlements follow the same timing logic as production, letting you test payout reconciliation flows end-to-end.
  </Card>
</CardGroup>

### Inspecting Sandbox Webhook Events

Point your webhook endpoint to a temporary inspection URL during development so you can see the full event payload without deploying a server.

```bash
# Register a webhook endpoint in the sandbox
curl -X POST https://api.paymatrix.com/v1/webhooks \
  -H "Authorization: Bearer pm_test_xxxxxxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://webhook.site/your-unique-id",
    "events": ["payment.succeeded", "payment.failed", "refund.created"]
  }'
```

<Tip>
  Bookmark [https://dashboard.paymatrix.com/sandbox](https://dashboard.paymatrix.com/sandbox) to view all test transactions, inspect request logs, and replay webhook events at any time.
</Tip>

## End-to-End Test Walkthrough

Follow these steps to verify your complete integration in the sandbox from order creation to webhook confirmation.

<Steps>
  ### Set Your API Key to a Test Key

  Confirm that every request your integration makes uses a key prefixed with `pm_test_`. Check your environment variables or configuration files to make sure no live key is present.

  ### Create an Order via the API or Dashboard

  Call `POST /v1/orders` with your test key, or create a test order directly from the sandbox dashboard at [dashboard.paymatrix.com/sandbox](https://dashboard.paymatrix.com/sandbox). Note the returned `orderId`.

  ### Submit a Test Card Number

  Use your integration's payment form (web checkout, mobile SDK, or direct API call) to submit a test card. Start with `4242 4242 4242 4242` for a straightforward success scenario, using any future expiry, any CVC, and any billing ZIP.

  ### Verify the Payment Status is `succeeded`

  Query the payment or order endpoint and confirm the `status` field reads `succeeded`:

  ```bash
  curl https://api.paymatrix.com/v1/payments/pay_xxxxxxxxxx \
    -H "Authorization: Bearer pm_test_xxxxxxxxxxxx"
  ```

  ```json
  {
    "id": "pay_xxxxxxxxxx",
    "status": "succeeded",
    "amount": 4999,
    "currency": "usd"
  }
  ```

  ### Confirm Your Webhook Endpoint Received `payment.succeeded`

  Check your webhook inspection URL (e.g. webhook.site) or your server logs to verify that PayMatrix delivered a `payment.succeeded` event with the correct `orderId` and `paymentId`. This confirms your fulfillment logic will trigger correctly in production.
</Steps>
